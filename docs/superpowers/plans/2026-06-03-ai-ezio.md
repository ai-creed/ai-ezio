# ai-ezio — implementation plan

- **Date:** 2026-06-03
- **Status:** ready for execution
- **Source spec:** `docs/superpowers/specs/2026-06-03-ai-ezio-design.md` (approved)
- **Reference docs:** `docs/architecture.md`, `docs/protocol.md`, `docs/milestones.md`, `UPSTREAM.md`
- **Canonical copy:** `~/.ai-pref-nsync/local-docs/ai-ezio/plans/` (this file is the synced mirror)

## Intent

Turn the approved design into a buildable, testable, milestone-by-milestone plan.
The end state: a single installable `ai-ezio` artifact that (a) runs hax for
humans with the terminal UI intact, and (b) is machine-controllable over an
explicit JSONL protocol on inherited fds, integrated into ai-whisper as a
first-class agent type.

This plan is **execution-ordered**. Each milestone lists its tasks, the exact
files/areas it touches, the engine-vs-harness boundary it must respect, and a
concrete "done when" with verification commands. Earlier milestones unblock
later ones; do not start a milestone until its predecessor's acceptance gate is
green.

## Ground rules (non-negotiable, from AGENTS.md + UPSTREAM.md)

1. **Engine work is C in `vendor/hax`; product work is TS in `packages/`.** When
   in doubt, push behavior into the harness.
2. **The hax patch stays tiny:** `src/protocol/emit.c` (+header), `--protocol-fd`
   / `--control-fd` flags, ~2–3 registration/flag lines, one `meson.build` line.
   Anything larger is a smell — re-home it in TS.
3. **Protocol is the contract.** No inferring readiness or response text from
   terminal chrome. Every new event/control is documented in `docs/protocol.md`
   *before* it is implemented.
4. **No scraping, ever.** stdout/stderr stay human-only.
5. **One installable artifact.** Never add a step that makes users install hax
   separately. Preserve npm-package + per-platform binary distribution.
6. **Targets are macOS + Linux (arm64 + x64).** Transport uses inherited fds;
   don't assume Windows fd semantics.
7. **Style split is intentional:** TS uses tabs / double quotes / semicolons /
   trailing commas; vendored C follows hax style (4-space indent, spaces,
   snake_case, `clang-format -i`). Never reformat hax sources to TS style.

## Current repo state (starting point)

- Docs-only: `README.md`, `NOTICE`, `LICENSE`, `UPSTREAM.md`, `AGENTS.md`,
  `docs/` (architecture, protocol, milestones, specs).
- **No `packages/` yet. No `vendor/hax` submodule yet. No `.gitmodules`.**
- `CLAUDE.md` is a symlink to `AGENTS.md`.
- So M1 starts from zero scaffolding: it must create the submodule wiring and the
  pnpm workspace from scratch.

## Architecture recap (so each task knows its lane)

```
packages/protocol   leaf: JSONL schema + codec + transport interface (fd impl)
packages/harness    owns hax child + session/turn lifecycle; typed API; controls
packages/adapter    ai-whisper handoff/handback glue (adapter-ai-ezio)
packages/cli        `ai-ezio` binary: REPL passthrough, --mount-mode, doctor, skills
vendor/hax          C engine submodule; emitter patch on `emitter` branch
```

Dependency direction: `protocol` ← `harness` ← {`adapter`, `cli`}. Keep it acyclic.

## M1 — Downstream foundation

**Goal:** one install produces a working `ai-ezio` that runs hax for humans, and
`--version --json` reports both versions.

### Tasks

1. **Vendor hax as a submodule.**
   - `git submodule add https://github.com/OleksandrChekhovskyi/hax vendor/hax`.
   - Pin to base commit `8fd139b5db49bd0b1d552c2530a18b547b3f4f4c` (per UPSTREAM.md).
   - Create the `emitter` branch in the submodule tracking `upstream/main`
     (empty for now; patch lands in M3). Add `hax-upstream` remote.
   - Commit the submodule pointer + `.gitmodules`.
2. **Build hax locally to validate the toolchain.**
   - `meson setup vendor/hax/build && meson compile -C vendor/hax/build`.
   - `meson test -C vendor/hax/build --print-errorlogs` (baseline green).
   - Document the toolchain prereqs (meson, ninja, C compiler, jansson) in a
     `docs/development.md` build section.
3. **Stand up the pnpm monorepo.**
   - Root `package.json` (private, workspaces), `pnpm-workspace.yaml`,
     `tsconfig.base.json`, `.npmrc`.
   - `packages/{protocol,harness,adapter,cli}` skeletons, each with
     `package.json`, `tsconfig.json` (project references), `src/index.ts`,
     `src/index.test.ts` placeholder.
   - Wire the dependency graph via workspace `*` deps + TS project references.
4. **Formatting/linting baseline (per AGENTS.md).**
   - ESLint (flat config) + Prettier + `.editorconfig`.
   - Baseline: **tabs**, **double quotes**, **semicolons**, **multiline trailing
     commas**, source-only formatting (exclude `vendor/hax`, build output, dist).
   - Add `format`, `lint`, `typecheck` scripts; ensure `vendor/hax` is excluded
     from every TS/JS tool's globs.
5. **Test runner.** Pick Vitest (fast, ESM-native, matches Node LTS). Root
   `test` script runs `pnpm -r test`. Add a single trivial passing test per
   package so CI is meaningful from day one.
6. **Binary resolver (`packages/harness` or a small shared `paths` module).**
   - Resolve hax binary in order: `AI_EZIO_HAX_BIN` → `require.resolve` of
     `@ai-ezio/hax-<os>-<cpu>` → `vendor/hax/build/hax` (dev fallback).
   - On miss: throw a clear error pointing at `ai-ezio doctor`.
   - Unit-test the resolver with a fake fs / env (all three branches + miss).
7. **`ai-ezio` CLI launcher (M1 scope = passthrough only).**
   - `packages/cli` `bin` entry: spawn the resolved hax with inherited
     stdio for interactive REPL passthrough; support `-p <prompt>` one-shot,
     both delegating straight to hax (no protocol yet).
   - `ai-ezio --version --json` → `{ ezioVersion, haxBaseCommit }` (read ezio
     version from package.json; hax base commit from a generated constant or
     `git -C vendor/hax rev-parse`).
8. **Packaging — host platform real, cross-compile matrix last.**
   - Main `ai-ezio` package declares `@ai-ezio/hax-<os>-<cpu>` as
     `optionalDependencies`, gated by `os`/`cpu`.
   - Stub the four platform packages (`darwin-arm64`, `darwin-x64`, `linux-x64`,
     `linux-arm64`) with their `os`/`cpu` fields.
   - **For the host platform, build the real binary into its package now** (copy
     `vendor/hax/build/hax` into `@ai-ezio/hax-<host-os>-<host-cpu>`). The other
     three platforms' binaries are filled by the CI cross-compile matrix later
     (it can land last). The dev fallback (#3 in resolver) covers day-to-day
     development, but M1 acceptance must NOT rely on it — see the install smoke
     test below.
9. **Single-install smoke test (proves the milestone contract).**
   - Pack the workspace (`pnpm pack` / `npm pack` of `ai-ezio` + the host
     platform binary package), install the tarballs into a **clean temp dir with
     no `vendor/hax` and `AI_EZIO_HAX_BIN` unset**, and run `ai-ezio` there.
   - This forces the resolver down the **platform-package** branch (not the dev
     fallback), proving "one install produces a working `ai-ezio`" with the hax
     binary embedded and no separate hax install. Script it so CI can run it for
     the host platform on every change.

### Files / areas

- New: `.gitmodules`, `vendor/hax` (submodule), `pnpm-workspace.yaml`,
  root `package.json`, `tsconfig.base.json`, `eslint.config.mjs`,
  `.prettierrc`, `.editorconfig`, `docs/development.md`.
- New: `packages/protocol`, `packages/harness`, `packages/adapter`,
  `packages/cli` skeletons; `packages/cli/bin/ai-ezio` launcher.
- New: `packaging/` platform package stubs (or `packages/hax-*`).

### Done when

- `pnpm install && pnpm -r build && pnpm -r test` are green.
- `meson compile -C vendor/hax/build` produces `vendor/hax/build/hax`.
- `ai-ezio` (dev-linked) launches an interactive hax REPL for a human.
- `ai-ezio -p "hello"` runs a one-shot through hax.
- `ai-ezio --version --json` emits both ezio version and hax base commit.
- **Single-install proof:** the packed `ai-ezio` installed into a clean temp dir
  (no `vendor/hax`, `AI_EZIO_HAX_BIN` unset) runs `ai-ezio --version --json` and
  a `-p` one-shot successfully — resolving the hax binary from the host
  `@ai-ezio/hax-<os>-<cpu>` package, with no separate hax install. This is the
  acceptance evidence for the milestone contract ("one install produces a working
  `ai-ezio`"); the dev fallback alone does not satisfy M1.

### Risks / notes

- This milestone touches >3 files by nature (scaffolding). It is pre-approved as
  a single foundational unit; still land it as a few reviewable commits
  (submodule wiring; workspace; tooling; CLI launcher) rather than one blob.

## M3 — Protocol MVP (includes the C emitter)

**Goal:** the harness drives a full turn over fds with no stdout scraping; the
human REPL is byte-for-byte unaffected when the protocol is off.

### Tasks

1. **Document any protocol deltas first** (per ground rule 3). The MVP event set
   is already in `docs/protocol.md`; if implementation needs a field tweak,
   update the doc before the code.
2. **Land the hax emitter patch on the `emitter` branch** (engine work, follow
   hax style + `clang-format -i`):
   - New `vendor/hax/src/protocol/emit.c` (+ header) hooking the existing
     `turn` `on_event(struct stream_event *)` callback; translate each
     `stream_event` (text delta, tool start/delta/end, reasoning, done, error)
     into the JSONL events. Use jansson (already linked) for encoding.
   - New CLI flags `--protocol-fd=<n>` / `--control-fd=<n>`.
   - ~2–3 lines to register the callback + parse flags; one `meson.build` line.
   - Read controls from the control fd and inject them into the **same input
     path the REPL uses** (do not fork a second input pipeline).
   - Build + test: `meson compile` / `meson test`. Verify `HAX_PROVIDER=mock`
     produces a deterministic event stream.
3. **`packages/protocol`:**
   - TypeScript types for every event and control in `docs/protocol.md`.
   - Encode/decode codec (JSONL line framing, UTF-8, `\n`-terminated, partial-line
     buffering on read).
   - Transport interface: `read(): AsyncIterable<Event>`, `send(control): void`.
   - **fd transport** implementation (fd 3 events in, fd 4 controls out).
   - Protocol version constant + a `ready.protocol` semver check helper.
4. **`packages/harness` lifecycle:**
   - Spawn hax with `--mount-mode --protocol-fd=3 --control-fd=4` and the pipes
     wired as fds 3/4; child dies with parent.
   - State machine: starting → idle → user_turn → assistant_turn → idle, with
     turn-scoped vs fatal error handling (per the protocol state diagram).
   - Typed session API: `ready`, turn started/finished, idle, error, last
     response. Treat `idle` as the only safe point to issue the next `submit`.
   - Reject driving an engine whose `protocol` major version is unsupported.
5. **Control path end to end for `submit` and `interrupt`.**
6. **Prove the REPL is unaffected when fds are absent** (snapshot/byte-compare a
   normal REPL session with protocol off).

### Engine boundary check

If the emitter starts needing more than the seam above (e.g. core refactors to
expose events), STOP and reconsider — that behavior likely belongs in the
harness, or warrants an upstream discussion. Keep `emit.c` anchored on
`on_event`.

### Files / areas

- New (C, `emitter` branch): `vendor/hax/src/protocol/emit.c` (+ header), flag
  parsing, registration, `meson.build` line. Bump submodule pointer in ai-ezio.
- New (TS): `packages/protocol/src/{events,controls,codec,transport-fd,version}.ts`;
  `packages/harness/src/{spawn,session,state-machine}.ts`.

### Done when

- With `HAX_PROVIDER=mock`, the harness submits a turn and observes
  `ready → user_turn_started → assistant_turn_started → assistant_turn_finished
  → idle` over fds, reading `assistant_turn_finished.content` as the handback —
  no stdout parsing anywhere.
- `interrupt` cancels an in-flight turn and returns to idle.
- A normal `ai-ezio` REPL run (no fds) is byte-for-byte identical to pre-patch.
- `meson test -C vendor/hax/build` and `pnpm -r test` green.

### Tests

- Protocol codec: round-trip every event/control; partial-line buffering;
  malformed-line handling.
- Harness state machine: deterministic with the mock provider; version-mismatch
  rejection; interrupt path; error → idle vs fatal.

## M4 — Mounted mode

**Goal:** a programmatic client submits and receives handback text purely over
the protocol — no clipboard, no scraping.

### Tasks

1. **`ai-ezio --mount-mode`:** protocol fds wired, REPL chrome suppressed
   (no spinner/prompt noise on stdout). stdout/stderr stay human-readable but
   quiet; control flow is entirely on the fds.
2. **Programmatic submit/handback:** accept `submit`, return final response via
   `assistant_turn_finished.content` through the harness API.
3. **`copy_last_response` (fd4 control → fd3 re-emit):** the harness sends the
   `copy_last_response` control over fd 4; hax re-emits the last
   `assistant_turn_finished` over fd 3. This is the documented contract in
   `docs/protocol.md` ("Controls (harness → hax, fd 4)") — implement it on the
   real fd control path so the handback can be re-fetched without re-running a
   turn and without clipboard access. **Do NOT shortcut this by re-emitting from
   a harness-side cache**: that would bypass the documented fd4 control and let
   an fd-control regression slip through. (If the team ever decides a
   harness-cache implementation is acceptable, the protocol doc must be updated
   first — protocol-is-the-contract; see ground rule 3.)
4. **`new_conversation`** (maps to hax `/new`) and **`status`** controls.
   Define the `status` payload shape now (model, context %, provider — tie to
   `--version --json` fields) and document it in `docs/protocol.md` before coding.
5. **Harness-level convenience API** for mounted consumers: `submitAndWait(text)
   → content`, plus `lastResponse()`, `newConversation()`, `status()`.

### Files / areas

- `packages/cli` mount-mode entry + chrome suppression.
- `packages/harness` control forwarding (sends `copy_last_response`,
  `new_conversation`, `status` over fd 4) + convenience API.
- `vendor/hax` (emitter seam): the engine consumes `copy_last_response`,
  `new_conversation`, and `status` from the control fd and re-emits / acts over
  fd 3, per `docs/protocol.md`. Keep these on the existing control-read path the
  emitter patch already adds (do not grow the patch beyond the seam). The
  harness-cache shortcut for `copy_last_response` is explicitly disallowed — it
  must traverse fd4 → hax → fd3.

### Done when

- A test client (Vitest, mock provider) runs:
  `submit → assistant_turn_finished.content`, then `copy_last_response` returns
  the same content without a new turn, then `new_conversation` resets, then
  `status` returns a payload — all over fds.

### Tests

- Mount-mode chrome suppression (no protocol bytes on stdout; no stdout on fd 3).
- **`copy_last_response` fd-control regression test:** assert the harness writes
  the `copy_last_response` control to fd 4 and that hax re-emits the last
  `assistant_turn_finished` over fd 3 with identical content and zero new turns
  (no `user_turn_started`/`assistant_turn_started`). The assertion must observe
  the fd traffic, not a harness-internal cache, so a cache-only shortcut fails
  the test.
- `new_conversation` (control over fd 4) clears last-response and conversation
  state, observed via fd 3.
- `status` control over fd 4 yields a `status` event/payload over fd 3.

## Cross-cutting concerns

### Testing strategy

- **Deterministic first:** every harness/protocol/adapter test uses
  `HAX_PROVIDER=mock` so there is no LLM round-trip. Real-provider runs are
  manual smoke checks at M5/M6 only.
- **Layered:** protocol (codec/transport units) → harness (state machine) →
  adapter (handoff round-trip) → CLI (launch/version/doctor) → e2e (mounted turn
  over fds with mock).
- **Engine:** `meson test -C vendor/hax/build` stays green across the emitter
  patch; add a small C/integration test that the emitter produces well-formed
  JSONL under the mock provider.
- **Per-feature TDD** (project preference): write the failing test first for each
  protocol event, control, and state transition.

### Debug hooks (use throughout)

- `HAX_PROVIDER=mock` — scripted provider for deterministic protocol/harness tests.
- `HAX_TRACE=<path>` — wire-level HTTP/SSE dump when debugging real providers.
- `HAX_TRANSCRIPT=<path>` — model-perspective transcript mirror.

### CI

- TS job: `pnpm install && pnpm -r build && pnpm lint && pnpm -r test`.
- Engine job: `meson setup/compile/test` on the patched submodule.
- **Cross-compile matrix** (darwin-arm64/x64, linux-x64/arm64) producing the
  per-platform hax binaries and copying each into its `@ai-ezio/hax-<os>-<cpu>`
  package. This is the one genuinely involved piece; it can land **last**, after
  M1–M4 prove out on the dev fallback resolver.

### Distribution invariants (guard every milestone)

- Always one install; never a separate hax install step.
- Resolver order preserved: `AI_EZIO_HAX_BIN` → platform package → dev fallback.
- The main package stays `import`-able (library use by ai-whisper) — never switch
  to a compiled single-file (Node SEA / `bun --compile`) that breaks `import`.

### Documentation upkeep

- Any protocol change: update `docs/protocol.md` first.
- Any locked-decision change: update `docs/architecture.md` and the spec.
- Keep `UPSTREAM.md`'s change-surface list accurate as the emitter lands.

## Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| Upstream redesign of hax's event model breaks `emit.c` | Keep the patch anchored on the stable `on_event` seam; localized re-port; upstream the emitter so the patch eventually disappears. |
| fd inheritance is Unix-only | Transport seam already abstracts the wire; add a stdio-framing transport if Windows is ever required — same JSONL schema. |
| Emitter patch creeps beyond the seam | Hard rule: anything past one file + two flags + ~3 lines + one meson line goes into the TS harness. Review the diff size at M3. |
| Skills sharing model unresolved | M2 study produces `docs/skills.md` and a decision before M6 depends on it. |
| Cross-compile matrix complexity | Defer to last; M1–M4 run on the dev-fallback resolver; ship the matrix once behavior is proven. |
| ai-whisper `AgentType` change is cross-repo | Coordinate the M6 change in ai-whisper; default ai-ezio to a replacement role to avoid the >2-agent question blocking. |

## Open questions to resolve during execution

- `assistant_delta` streaming opt-in (a control to enable streaming) vs always-on
  — decide at M3 based on fd traffic; document in `docs/protocol.md`.
- Whether `tool_call_delta` is needed for long-running tool output — defer until
  M4/M5 surface a real need.
- `status` event payload shape — pin at M4 alongside `--version --json` fields.
- >2 mounted agents in ai-whisper vs ai-ezio as a replacement role — M6.
- Skills sharing model across Claude/Codex/hax/ai-ezio — M2.
- Which generic improvements (esp. the emitter) to upstream — ongoing per
  `UPSTREAM.md` merge policy.

## Suggested execution order (summary)

1. M1 foundation (submodule + workspace + tooling + CLI passthrough + resolver).
2. M2 skills (human-usable product early; informs M6) — no C changes.
3. M3 protocol MVP (the one C patch + protocol/harness over fds).
4. M4 mounted mode (full control set; clipboard-free handback).
5. M5 adapter (single relay handoff via protocol).
6. M6 workflow integration (first-class ai-whisper agent type).
7. Cross-compile publish matrix (can finish last).
